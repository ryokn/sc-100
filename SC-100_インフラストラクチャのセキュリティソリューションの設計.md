# SC-100: インフラストラクチャのセキュリティソリューションの設計

> **対象資格**: Microsoft サイバーセキュリティ アーキテクト (SC-100)  
> **レベル**: 上級  
> **試験ウェイト**: 25〜30%  
> **元ラーニングパス**: [Microsoft Learn](https://learn.microsoft.com/ja-jp/training/paths/sc-100-design-security-solutions-infrastructure/)

## ラーニングパスの概要

このラーニングパスでは、インフラストラクチャセキュリティのソリューションを設計する方法を学習します。対象範囲には、さまざまなクラウドモデル（SaaS、PaaS、IaaS）のセキュリティ要件の指定、ハイブリッド環境・マルチクラウド環境でのセキュリティ態勢管理、エンドポイントの保護、ネットワークセキュリティが含まれます。

### 前提条件

- ID とアクセス、プラットフォーム保護、セキュリティ運用、データ・アプリケーションのセキュリティに関する高度な知識と実務経験
- ハイブリッド実装とクラウド実装の経験

### 1-1. クラウドの共有責任モデル

クラウドサービスモデルによって、セキュリティ責任の分担が異なります。

```
                オンプレミス    IaaS      PaaS      SaaS
アプリケーション [お客様]    [お客様]  [お客様]  [プロバイダー]
データ          [お客様]    [お客様]  [お客様]  [お客様/共有]
ランタイム       [お客様]    [お客様]  [プロバイダー][プロバイダー]
ミドルウェア     [お客様]    [お客様]  [プロバイダー][プロバイダー]
OS              [お客様]    [お客様]  [プロバイダー][プロバイダー]
仮想化          [お客様]    [プロバイダー][プロバイダー][プロバイダー]
サーバー         [お客様]    [プロバイダー][プロバイダー][プロバイダー]
ネットワーク     [お客様]    [プロバイダー][プロバイダー][プロバイダー]
ストレージ       [お客様]    [プロバイダー][プロバイダー][プロバイダー]
```

---

### 1-2. IaaS のセキュリティ要件

IaaS では OS・ミドルウェア・アプリケーションの管理責任がお客様にあります。

#### Azure 仮想マシンのセキュリティ

| セキュリティ領域 | 要件・推奨事項 |
|---|---|
| **OS の強化** | ベースラインセキュリティ設定の適用（CIS ベンチマーク等） |
| **パッチ管理** | Azure Update Manager による定期的な更新プログラムの適用 |
| **エンドポイント保護** | Microsoft Defender for Endpoint のオンボーディング |
| **ネットワーク制御** | NSG (Network Security Group) によるトラフィック制御 |
| **リモートアクセス** | Azure Bastion を使用したパブリック RDP/SSH の排除 |
| **ディスク暗号化** | Azure Disk Encryption による OS・データディスクの暗号化 |
| **特権アクセス** | JIT VM アクセスによる管理ポートのジャストインタイム開放 |

#### JIT (Just-In-Time) VM アクセス

Microsoft Defender for Cloud の JIT VM アクセスは、VM への管理ポート（RDP ポート 3389、SSH ポート 22 など）を必要な時だけ開放し、それ以外の時間は閉鎖する機能です。

**JIT のワークフロー:**
1. 管理者がアクセスリクエストを送信（対象IP、ポート、時間を指定）
2. Defender for Cloud が正当性を評価
3. 承認された場合、指定した時間だけ NSG ルールを追加
4. 時間切れ後、アクセスルールを自動削除

---

### 1-3. PaaS のセキュリティ要件

PaaS では OS やランタイムの管理はプロバイダーが担当しますが、アプリケーションとデータの保護はお客様の責任です。

#### Azure App Service のセキュリティ

| セキュリティ設定 | 説明 |
|---|---|
| **認証/認可の有効化** | App Service の組み込み認証を使用して Entra ID と連携 |
| **HTTPS のみを強制** | HTTP アクセスを HTTPS にリダイレクト |
| **最小 TLS バージョンの設定** | TLS 1.2 以上を要求 |
| **プライベートエンドポイント** | パブリック インターネットからのアクセスを遮断 |
| **VNet 統合** | App Service を仮想ネットワークに接続 |
| **マネージド ID** | シークレットなしで Azure サービスにアクセス |

#### Azure Functions・Kubernetes Service のセキュリティ

**Azure Functions (サーバーレス):**
- 関数アプリへのアクセスを Entra ID 認証で保護
- 機密情報は環境変数ではなく Key Vault 参照を使用
- アウトバウンド VNet 統合でネットワークを制御

**Azure Kubernetes Service (AKS):**
- RBAC とネームスペースによるアクセス分離
- マネージド ID を使用したクラスタの ID 管理
- ネットワークポリシーによるポッド間通信の制御
- Microsoft Defender for Containers によるランタイム保護
- コンテナイメージの脆弱性スキャン（Azure Container Registry との統合）

---

### 1-4. SaaS のセキュリティ要件

SaaS では基盤となるインフラはプロバイダーが管理しますが、データとアクセス管理はお客様の責任です。

**SaaS セキュリティの主な考慮事項:**

| 観点 | 要件 |
|---|---|
| **ID 統合** | シングルサインオン (SSO) と条件付きアクセスの統合 |
| **データ保護** | SaaS アプリに保存されるデータの暗号化と分類 |
| **アクセス制御** | 最小権限に基づくロールの設定 |
| **Shadow IT** | 未承認 SaaS アプリの検出と制御 |
| **コンプライアンス** | SaaS プロバイダーのコンプライアンス証明書の確認 |
| **セッション制御** | Microsoft Defender for Cloud Apps によるリアルタイム制御 |

---

### 1-5. IoT・OT のセキュリティ要件

IoT（モノのインターネット）と OT（オペレーショナルテクノロジー）は、従来の IT セキュリティとは異なる課題を持ちます。

**IT と OT のセキュリティの違い:**

| 観点 | IT セキュリティ | OT セキュリティ |
|---|---|---|
| **最優先事項** | 機密性・整合性・可用性 (CIA) | 安全性・可用性・整合性 |
| **パッチ適用** | 定期的な更新が可能 | 停止できないため困難 |
| **デバイスライフサイクル** | 数年〜10年 | 10年〜数十年 |
| **プロトコル** | 標準 TCP/IP | Modbus、PROFINET 等の特殊プロトコル |

**Microsoft Defender for IoT:**

- エージェントレスのネットワーク監視（パッシブな資産発見）
- IoT/OT 特有のプロトコルの深いパケット検査
- 異常動作の検出と脅威インテリジェンスの統合
- Purdue モデルに基づくネットワーク階層の可視化

---

### 1-6. コンテナと AI ワークロードのセキュリティ

#### コンテナセキュリティ

**コンテナのセキュリティのライフサイクル:**

| フェーズ | セキュリティ対策 |
|---|---|
| **ビルド** | ベースイメージの脆弱性スキャン、Dockerfile のセキュリティレビュー |
| **レジストリ** | プライベートレジストリの使用、ACR タスクでの自動スキャン |
| **デプロイ** | 署名されたイメージのみを許可（イメージ完全性） |
| **ランタイム** | コンテナの動作監視、異常検出、ポッドセキュリティ標準 |

#### AI ワークロードのセキュリティ

AI/ML ワークロード固有のセキュリティ要件:

- **訓練データの保護**: 学習データの完全性と機密性を確保
- **モデルの保護**: モデルの盗用・改ざんを防止
- **推論エンドポイントのセキュリティ**: API への認証・認可・レート制限
- **Azure AI Content Safety**: 有害コンテンツのフィルタリング
- **Microsoft Purview によるデータガバナンス**: AI が使用するデータの管理

## モジュール 2: ハイブリッドおよびマルチクラウド環境でのセキュリティ態勢管理のためのソリューションを設計する

### 学習目標

- Microsoft Defender for Cloud を使用してセキュリティ態勢管理ソリューションを設計できる
- Azure Arc を活用してハイブリッド・マルチクラウド環境に統合できる
- Microsoft Cloud Security Benchmark (MCSB) を適用できる

---

### 2-1. セキュリティ態勢管理の概念

**CSPM (Cloud Security Posture Management)** は、クラウドリソースのセキュリティ設定を継続的に評価・改善するアプローチです。

**CSPM の主な機能:**

| 機能 | 説明 |
|---|---|
| **セキュリティ態勢の可視化** | セキュアスコアによる定量的な評価 |
| **推奨事項** | 優先度付きの具体的な改善アクション |
| **アタックパス分析** | 攻撃者が悪用できる経路を可視化 |
| **規制コンプライアンス** | MCSB、CIS、PCI DSS などへの準拠評価 |
| **エクスポージャー管理** | インターネットに露出しているリソースの特定 |

---

### 2-2. Microsoft Defender for Cloud

Defender for Cloud は、マルチクラウド・ハイブリッド環境全体の CSPM と CWPP (Cloud Workload Protection Platform) を提供します。

#### Defender for Cloud の 2 つの柱

**1. CSPM（クラウドセキュリティ態勢管理）:**
- セキュアスコア: 0〜100 のスコアで現在のセキュリティ態勢を評価
- セキュリティ推奨事項: 設定ミスや脆弱性を検出し、修正手順を提供
- アタックパス分析: 攻撃者が最も悪用しやすい経路を視覚化
- クラウドセキュリティグラフ: 環境全体のリソースと関係を把握

**2. CWPP（クラウドワークロード保護プラットフォーム）:**

| Defender プラン | 保護対象 |
|---|---|
| **Defender for Servers** | Azure VM・AWS EC2・GCP Compute Engine・オンプレミスサーバー |
| **Defender for SQL** | Azure SQL・SQL Server（VM上・オンプレミス） |
| **Defender for Storage** | Azure Blob Storage・Azure Files・Azure Data Lake |
| **Defender for Containers** | AKS・Arc 対応 Kubernetes |
| **Defender for App Service** | Azure App Service |
| **Defender for Key Vault** | Azure Key Vault への不審なアクセス検出 |
| **Defender for ARM** | Azure Resource Manager へのリスクの高い操作を検出 |

---

### 2-3. Azure Arc によるハイブリッド・マルチクラウドの統合

Azure Arc は、Azure 以外の場所（オンプレミス、他のクラウド）にあるリソースを Azure の管理フレームワークで統合管理するサービスです。

**Azure Arc が対応するリソース:**

| リソース種別 | 説明 |
|---|---|
| **Arc 対応サーバー** | Windows/Linux のオンプレミスまたは他クラウドのサーバー |
| **Arc 対応 Kubernetes** | EKS、GKE、オンプレミスの Kubernetes クラスター |
| **Arc 対応 SQL Server** | オンプレミスの SQL Server |
| **Arc 対応データサービス** | オンプレミスで Azure SQL・PostgreSQL を実行 |

**Arc によるセキュリティの統合:**

- Defender for Cloud の CSPM と CWPP をオンプレミスリソースに適用
- Azure Policy でオンプレミスのサーバーにセキュリティポリシーを強制
- Azure Monitor でオンプレミスからログを収集
- Defender for Endpoint を Arc サーバーにデプロイ

---

### 2-4. Microsoft Cloud Security Benchmark (MCSB) の適用

MCSB は、Azure とマルチクラウド環境向けのセキュリティベストプラクティスの集大成です。Defender for Cloud を通じて自動評価されます。

**MCSB と他フレームワークのマッピング:**

MCSB は以下の業界標準フレームワークとの対応関係が文書化されています。

- CIS Benchmarks
- NIST SP 800-53
- PCI DSS
- ISO 27001

これにより、MCSB の推奨事項を実装することで、複数の規制への準拠を同時に進められます。

## モジュール 3: サーバーとクライアントのエンドポイントをセキュリティで保護するためのソリューションを設計する

### 学習目標

- サーバー・クライアント・IoT・OT・モバイル・組み込みデバイスのセキュリティ要件を分析できる

---

### 3-1. エンドポイントセキュリティの設計原則

**エンドポイントセキュリティの多層防御:**

```
┌─────────────────────────────────────────────────────┐
│  ポリシーと構成管理 (Intune / Azure Arc)             │
├─────────────────────────────────────────────────────┤
│  脅威検出と対応 (Microsoft Defender for Endpoint)    │
├─────────────────────────────────────────────────────┤
│  ID とアクセス管理 (条件付きアクセス / デバイスコンプライアンス)│
├─────────────────────────────────────────────────────┤
│  OS 強化 (セキュリティベースライン / CIS ベンチマーク)  │
└─────────────────────────────────────────────────────┘
```

---

### 3-2. Microsoft Defender for Endpoint

Microsoft Defender for Endpoint (MDE) は、企業向けのエンドポイント検出・対応 (EDR) ソリューションです。

**MDE の主な機能:**

| 機能 | 説明 |
|---|---|
| **脅威と脆弱性の管理 (TVM)** | 脆弱性・設定ミスを継続的に評価し、リスクスコアで優先順位付け |
| **攻撃対象領域の縮小 (ASR)** | マルウェアが悪用するベクターを制限するルール |
| **次世代保護** | AI 駆動のマルウェア対策・ランサムウェア対策 |
| **EDR (Endpoint Detection and Response)** | 高度な脅威の検出・調査・対応 |
| **自動調査と対応 (AIR)** | アラートの自動調査と封じ込めアクション |
| **Microsoft Secure Score for Devices** | デバイスのセキュリティ態勢スコア |

**サポートされるプラットフォーム:**
- Windows 10/11、Windows Server
- macOS
- Linux
- Android、iOS

---

### 3-3. エンドポイント構成管理

#### Microsoft Intune

Intune は、モバイルデバイス管理 (MDM) とモバイルアプリ管理 (MAM) を提供するクラウドベースのサービスです。

**Intune の主な機能:**

| 機能 | 説明 |
|---|---|
| **デバイス登録** | Windows・macOS・iOS・Android を統一管理 |
| **コンプライアンスポリシー** | デバイスが満たすべきセキュリティ要件を定義 |
| **構成プロファイル** | デバイス設定（Wi-Fi、VPN、証明書など）を一元配布 |
| **アプリの展開** | ビジネスアプリを承認済みデバイスに展開 |
| **条件付きアクセスとの連携** | コンプライアンス違反デバイスのアクセスをブロック |
| **Windows Autopilot** | 新しい Windows デバイスをゼロタッチで展開 |

#### セキュリティベースライン

Microsoft は、Windows・Edge・MDE などのセキュリティ推奨設定をまとめた**セキュリティベースライン**を提供しています。Intune または グループポリシーで適用できます。

**主なセキュリティベースライン:**

- **Windows セキュリティベースライン**: Windows 10/11 の推奨設定
- **Microsoft 365 Apps ベースライン**: Office アプリの推奨設定
- **Microsoft Edge ベースライン**: ブラウザのセキュリティ設定
- **Microsoft Defender for Endpoint ベースライン**: MDE の推奨設定

### 3-4. 特権アクセスワークステーション (PAW) の設計

管理タスク専用の高セキュリティデバイス。詳細はラーニングパス 2 モジュール 3 を参照。

**PAW 展開の段階的アプローチ:**

| 段階 | 対象 | 内容 |
|---|---|---|
| **段階 1** | グローバル管理者 | 最高権限アカウントの PAW からのみ操作を許可 |
| **段階 2** | 全管理者 | Tier 1・Tier 2 の管理者も PAW を使用 |
| **段階 3** | 全特権アクセス | すべての特権アクセスを PAW に移行 |

---

### 3-5. リモートアクセスとエンドポイントセキュリティ戦略

#### Azure Bastion

Azure Bastion は、パブリック IP アドレスを VM に付与せず、Azure Portal から直接 RDP/SSH 接続するためのサービスです。

**利点:**
- VM にパブリック IP が不要（アタックサーフェスの縮小）
- RDP/SSH ポートをインターネットに公開しない
- マルウェアに対して安全なブラウザベースのセッション

#### Azure VPN Gateway / ExpressRoute

| 接続方法 | 適用場面 |
|---|---|
| **P2S VPN (Point-to-Site)** | 個別のリモートユーザーが Azure VNet に接続 |
| **S2S VPN (Site-to-Site)** | オンプレミスと Azure VNet をインターネット経由で接続 |
| **ExpressRoute** | オンプレミスと Azure を専用回線で接続（高可用性・大帯域） |
| **Azure Virtual WAN** | 大規模なブランチオフィスとのハブアンドスポーク接続 |

### 4-1. ネットワークセグメンテーションの設計

ゼロトラストの「侵害を想定する」原則に基づき、ネットワークセグメンテーションは攻撃者の横展開を制限するための重要な手法です。

#### Azure 仮想ネットワーク (VNet) の設計

**VNet セグメンテーションの基本構造:**

```
┌─────────────────────────────────────────────────────┐
│  Azure Virtual Network (VNet)                        │
│                                                     │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐│
│  │ Web サブネット│  │ App サブネット│  │ DB サブネット ││
│  │ (DMZ)        │  │              │  │              ││
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘│
│         │NSG              │NSG              │NSG     │
└─────────────────────────────────────────────────────┘
```

**ネットワークセキュリティグループ (NSG):**

NSG は、Azure リソースへのインバウンド・アウトバウンドネットワークトラフィックをフィルタリングするルールの集合です。

**NSG の設計ベストプラクティス:**

- デフォルト拒否 (Default Deny) の原則: 明示的に許可したトラフィックのみを通す
- サブネットと NIC の両方に NSG を適用する
- Application Security Group (ASG) を使用してワークロード単位でグループ化する
- NSG フローログを有効にして通信パターンを可視化する

#### マイクロセグメンテーション

マイクロセグメンテーションは、ネットワークを細かく分割することで、侵害が発生した場合の横展開を最小化します。

**Azure でのマイクロセグメンテーションの実装方法:**

1. **NSG のサブネット単位適用**: ワークロードごとに専用サブネットを作成し、NSG で制御
2. **Azure Firewall のポリシー**: アプリケーションルールとネットワークルールで詳細に制御
3. **Azure Virtual Network Manager**: 複数の VNet にわたる一元的なネットワークポリシー管理

---

### 4-2. トラフィックフィルタリング

#### Azure Firewall

Azure Firewall は、Azure VNet リソースを保護するためのクラウドネイティブのネットワークセキュリティサービスです。

**Azure Firewall の主な機能:**

| 機能 | 説明 |
|---|---|
| **FQDN フィルタリング** | ドメイン名（完全修飾ドメイン名）に基づくフィルタリング |
| **アプリケーションルール** | HTTP/HTTPS トラフィックのきめ細かい制御 |
| **ネットワークルール** | IP アドレス・ポート・プロトコルベースのフィルタリング |
| **DNAT ルール** | インバウンドトラフィックの内部リソースへの転送 |
| **脅威インテリジェンス** | 既知の悪意ある IP とドメインをブロック |
| **TLS インスペクション** | 暗号化された HTTPS トラフィックの検査 |
| **IDPS** | 侵入検出・防止システム（Firewall Premium） |

#### Web Application Firewall (WAF)

WAF は、Web アプリケーションを一般的な Web 攻撃から保護するサービスです。

**Azure の WAF ソリューション:**

| サービス | 説明 |
|---|---|
| **Azure Application Gateway WAF** | Web アプリケーションの前段に配置するリージョンスコープの WAF |
| **Azure Front Door WAF** | グローバルに分散したアプリケーションのエッジで保護 |
| **Azure CDN WAF** | CDN を経由したコンテンツ配信の保護 |

**WAF が防ぐ一般的な攻撃（OWASP Top 10）:**

- SQL インジェクション
- クロスサイトスクリプティング (XSS)
- クロスサイトリクエストフォージェリ (CSRF)
- 機密データの露出
- セキュリティの設定ミス

#### DDoS 保護

| 保護レベル | 説明 | 適用場面 |
|---|---|---|
| **DDoS Network Protection (Basic)** | Azure インフラの基本的な DDoS 対策 | 自動適用 |
| **DDoS Network Protection (Standard)** | 高度な DDoS 軽減・コスト保護・トレーニング | 重要なパブリック IP に推奨 |
| **DDoS IP Protection** | 個別の IP アドレス単位での保護 | 少数のパブリック IP を保護 |

### 4-3. ネットワーク監視

#### Azure Network Watcher

Azure Network Watcher は、Azure ネットワークの監視・診断・ログ収集を提供するサービスです。

**主な機能:**

| 機能 | 説明 |
|---|---|
| **接続モニター** | エンドポイント間の接続状態を継続的に監視 |
| **NSG フローログ** | NSG を通過したトラフィックをログ記録 |
| **パケットキャプチャ** | VM のネットワークトラフィックを一時的にキャプチャ |
| **IP フロー確認** | 特定のパケットが NSG ルールで許可/拒否されるかを確認 |
| **トポロジー** | VNet の構成を視覚的に表示 |

#### Microsoft Sentinel によるネットワーク監視

Sentinel のデータコネクタを通じて、NSG フローログや Azure Firewall ログを収集し、以下を実現します。

- 異常なネットワークトラフィックパターンの検出
- C2（コマンド&コントロール）通信の検出
- データ流出の兆候の検知
- ネットワークへの不審なアクセスに対するアラート

---

### 4-4. ネットワーク設計のベストプラクティス

#### プライベートエンドポイントと Private Link

**Azure Private Endpoint** は、VNet 内のプライベート IP アドレスを通じて Azure サービスに安全にアクセスするためのネットワークインターフェースです。

**Private Endpoint のメリット:**
- Azure サービスへのトラフィックがパブリックインターネットを経由しない
- サービスのパブリックアクセスを無効化できる
- オンプレミスから ExpressRoute/VPN 経由でもプライベートにアクセス可能

**Private Endpoint を推奨するサービス:**
- Azure Storage（Blob、File、Queue、Table）
- Azure SQL Database
- Azure Key Vault
- Azure Container Registry
- Azure App Service

#### Service Endpoint vs Private Endpoint

| 比較項目 | Service Endpoint | Private Endpoint |
|---|---|---|
| **IP アドレス** | サービスのパブリック IP を使用 | プライベート IP を割り当て |
| **ネットワーク経由** | Microsoft バックボーン（インターネット不使用） | 完全に VNet 内 |
| **オンプレミスアクセス** | 不可 | 可（ExpressRoute/VPN 経由） |
| **セキュリティ** | 高い | より高い |
| **コスト** | 無料 | Private Link の料金が発生 |

## モジュール 5: 対話型ケーススタディ — エンドポイントとインフラストラクチャのセキュリティ保護

### シナリオの概要

エンドポイントとインフラストラクチャのセキュリティに重点を置いた実際のビジネスシナリオに対して、サイバーセキュリティアーキテクトとしてのスキルを適用します。

### 評価される設計スキル

**クラウドセキュリティ態勢管理:**
- Defender for Cloud を使用したマルチクラウド環境の統合評価
- セキュアスコアの向上計画の策定
- アタックパス分析の活用

**エンドポイント保護:**
- Intune を使用したデバイス管理と構成の設計
- Defender for Endpoint の展開と設定
- セキュリティベースラインの適用計画

**ネットワークセキュリティ:**
- ハブアンドスポークネットワークアーキテクチャの設計
- NSG と Azure Firewall を組み合わせた多層防御
- Private Endpoint によるサービスアクセスのセキュリティ強化

## まとめ: ラーニングパス全体の重要ポイント

### 設計領域と主要ソリューションの対応

| 設計領域 | 主要な Microsoft ソリューション |
|---|---|
| クラウドサービス (SaaS/PaaS/IaaS) | 共有責任モデル、JIT VM アクセス、マネージド ID |
| セキュリティ態勢管理 | Defender for Cloud (CSPM/CWPP)、Azure Arc、MCSB |
| エンドポイント保護 | Defender for Endpoint、Intune、セキュリティベースライン |
| ネットワークセキュリティ | Azure Firewall、NSG、WAF、DDoS Protection、Private Endpoint |

### 重要な設計原則

1. **共有責任の明確化**: クラウドサービスモデル（SaaS/PaaS/IaaS）ごとにお客様の責任範囲を正確に理解する
2. **ハイブリッド統合**: Azure Arc と Defender for Cloud を組み合わせ、オンプレミスとクラウドを一元管理する
3. **デフォルト拒否**: ネットワークのデフォルトはすべてのトラフィックを拒否し、必要なものだけを明示的に許可する
4. **プライベートネットワークの活用**: Private Endpoint と Private Link を使用してデータがパブリックインターネットを経由しないようにする
5. **継続的な態勢評価**: Defender for Cloud のセキュアスコアと推奨事項を定期的にレビューし、継続的に改善する

### 試験での重要なポイント

- **JIT VM アクセス**は Defender for Cloud の機能であり、管理ポートをジャストインタイムで開放する
- **Azure Arc** はオンプレミスや他クラウドのリソースを Azure 管理フレームワークに統合する
- **Service Endpoint と Private Endpoint の違い**を明確に区別できるようにする
- **NSG はデバイスレベルとサブネットレベルの両方**に適用できる
- **マネージド ID の優先使用**がシークレット管理のベストプラクティスである
