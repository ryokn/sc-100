## ラーニングパスの概要

このラーニングパスでは、セキュリティオペレーション (SecOps)、ID とアクセス管理、特権アクセスの保護、および規制コンプライアンスに関するソリューションを設計する方法を学習します。

### 前提条件

- ID とアクセス、プラットフォーム保護、セキュリティ運用、データ・アプリケーションのセキュリティに関する高度な知識と実務経験
- ハイブリッド実装とクラウド実装の経験

## モジュール 1: 規制コンプライアンスのソリューションを設計する

### 学習目標

- コンプライアンス要件をセキュリティソリューションに変換できる
- Microsoft Purview を活用してコンプライアンスに対応できる
- Microsoft Priva でプライバシー要件に対処するソリューションを設計できる

### 1-1. コンプライアンス要件をセキュリティソリューションに変換する

#### 規制要件の解釈と変換

組織はさまざまな規制・標準への準拠を求められます。サイバーセキュリティアーキテクトは、これらの法的・規制的要件を具体的な技術コントロールへと変換する責任を担います。

**主な規制・標準フレームワーク:**

| フレームワーク | 対象・特徴 |
|---|---|
| **ISO 27001** | 情報セキュリティマネジメントシステム (ISMS) の国際標準 |
| **NIST CSF** | 米国国立標準技術研究所のサイバーセキュリティフレームワーク |
| **PCI DSS** | クレジットカード情報を扱う組織向けのセキュリティ標準 |
| **GDPR** | EU の一般データ保護規則（個人データの保護） |
| **HIPAA** | 米国の医療情報プライバシー法 |
| **FedRAMP** | 米国連邦政府クラウドサービスのセキュリティ標準 |
| **SOC 2** | サービス組織のセキュリティ・可用性・機密性などを評価 |

#### コンプライアンスの変換プロセス

```
規制要件の特定
     ↓
ギャップ分析（現状 vs 要件）
     ↓
コントロールへのマッピング
     ↓
技術的実装の設計
     ↓
継続的な監視と評価
     ↓
証拠の収集と報告
```

---

### 1-2. Microsoft Purview によるコンプライアンス対応

Microsoft Purview は、データガバナンス、リスク管理、コンプライアンスを一元管理するプラットフォームです。

#### Microsoft Purview Compliance Manager

Compliance Manager は、マルチクラウド環境全体の規制コンプライアンス要件を管理するためのツールです。

**主な機能:**

- **コンプライアンス スコア**: 現在のコンプライアンス態勢を定量的に評価（0〜100点）
- **改善アクション**: 優先度付きの具体的な改善手順を提供
- **コントロールマッピング**: 異なる規制フレームワーク間のコントロール対応を管理
- **AI ガバナンス**: AI システムに関するコンプライアンス要件への対応
- **マルチクラウドサポート**: Azure、Microsoft 365、AWS、GCP のコントロールを統合管理

**対応している主な規制標準:**

- Microsoft クラウドセキュリティベンチマーク (MCSB)
- GDPR、HIPAA、PCI DSS
- ISO 27001、ISO 27018
- NIST 800-53、NIST CSF
- SOC 2

#### Microsoft Purview Audit（監査）

Purview Audit は、組織全体のユーザーとサービスのアクティビティを記録・検索するための監査ソリューションです。

| 機能 | 説明 |
|---|---|
| **統合監査ログ** | Microsoft 365 サービス全体のアクティビティを一元収集 |
| **高度な監査 (Audit Premium)** | 重要なイベントの長期保持（最大 10 年）と帯域幅の増加 |
| **監査ログの保持ポリシー** | ログの保持期間を要件に合わせてカスタマイズ |
| **法的調査のサポート** | インシデント調査や法的要求に対応する証拠収集 |

---

### 1-3. Microsoft Priva によるプライバシー要件への対応

Microsoft Priva は、組織のプライバシーリスクを管理し、データ主体の権利要求 (DSR) に対応するためのソリューションです。

**主なコンポーネント:**

1. **Privacy Risk Management**: 個人データの過剰な露出、不適切な転送などのプライバシーリスクを検出・対処
2. **Subject Rights Requests**: GDPR などの規制に基づくデータ主体の権利要求（アクセス権、削除権など）を効率的に処理
3. **Consent Management**: ユーザーの同意設定を管理・追跡

---

### 1-4. Azure Policy によるコンプライアンスの実装

Azure Policy は、Azure リソースのルール・制約を定義・適用・評価するためのサービスです。

**ポリシー設計のベストプラクティス:**

- **イニシアティブ（ポリシーセット）の活用**: 複数のポリシーをグループ化して一括管理
- **管理グループへの適用**: サブスクリプション横断でポリシーを統一適用
- **効果（Effect）の選択**:
  - `Audit`: コンプライアンス違反を記録するが強制はしない
  - `Deny`: 非準拠リソースのデプロイを拒否
  - `DeployIfNotExists`: 必要なリソースが存在しない場合に自動デプロイ
  - `Modify`: リソースのプロパティを自動修正
- **カスタムポリシーの作成**: 組織固有の要件に対応するポリシーを定義

---

### 1-5. Microsoft Defender for Cloud によるコンプライアンス評価

Defender for Cloud の**規制コンプライアンスダッシュボード**は、選択した規制標準に対する現在の準拠状況を可視化します。

**主な機能:**

- リアルタイムのコンプライアンス状態の評価
- 各コントロールの詳細な準拠・非準拠状態
- 非準拠リソースの特定と修復ガイダンス
- コンプライアンスレポートのエクスポート（PDF、CSV）
- 規制標準のカスタム追加

## モジュール 2: ID およびアクセス管理のソリューションを設計する

### 学習目標

- さまざまなデプロイモデル（ハイブリッド、マルチクラウド）のアクセス戦略を設計できる
- 安全な外部コラボレーションを実現するソリューションを設計できる
- 先進認証を実装するソリューションを設計できる
- ID インフラストラクチャを保護するソリューションを設計できる

---

### 2-1. ID 戦略の設計

#### ハイブリッド・マルチクラウド環境の ID

現代の組織では、オンプレミス・クラウド・マルチクラウドにわたって ID を統合管理する必要があります。

**ID アーキテクチャのパターン:**

| パターン | 説明 | 適用場面 |
|---|---|---|
| **クラウドネイティブ** | Microsoft Entra ID のみ | 新規クラウド専用組織 |
| **ハイブリッド（同期）** | Entra Connect で AD DS と同期 | オンプレミス AD DS を持つ組織 |
| **フェデレーション** | AD FS や外部 IdP との連携 | 既存フェデレーション環境 |
| **External ID** | B2B・B2C でゲストや外部ユーザーを管理 | パートナーや顧客向けアクセス |

#### Microsoft Entra ID（旧 Azure Active Directory）

Microsoft Entra ID は Microsoft のクラウドベース ID・アクセス管理サービスです。

**コアコンポーネント:**

- **ユーザーとグループ**: クラウドネイティブ・同期・ゲストユーザーの管理
- **アプリケーション登録**: SaaS・カスタムアプリへのシングルサインオン (SSO)
- **条件付きアクセス**: リスクに基づく動的なアクセス制御
- **ID 保護 (Entra ID Protection)**: リスクの高いサインインや侵害された資格情報の検出
- **アクセスレビュー**: 定期的な権限の見直しと棚卸し

#### Active Directory Domain Services (AD DS) のセキュリティ強化

オンプレミスの AD DS は多くの組織の ID 基盤ですが、同時に攻撃者の主要標的でもあります。

**AD DS セキュリティのベストプラクティス:**

- **Tier モデルの実装**: 管理資産を Tier 0（ドメインコントローラー）、Tier 1（サーバー）、Tier 2（ワークステーション）に分離
- **特権アカウントの保護**: 管理アカウントと通常業務アカウントを分離
- **AD DS の回復性**: DR 計画、定期的なバックアップ、回復演習
- **一般的な攻撃への耐性**: Pass-the-Hash、Kerberoasting、DCSync 攻撃への対策

---

### 2-2. 条件付きアクセスの設計

条件付きアクセスは、ゼロトラストの「明示的に確認する」原則を実装するための中核的なメカニズムです。

#### 条件付きアクセスポリシーの設計要素

```
誰が (ユーザー・グループ・ロール)
     +
何に (アプリケーション・リソース)
     +
どこから (場所・デバイス・リスク)
     ↓
ポリシーエンジンが評価
     ↓
アクセスを許可 / MFA を要求 / 準拠デバイスを要求 / ブロック
```

**代表的なポリシーパターン:**

| シナリオ | ポリシー設定 |
|---|---|
| 管理者アカウント保護 | 管理者ロールに MFA を必須化 |
| レガシー認証のブロック | レガシー認証プロトコルを使用するアクセスをブロック |
| リスクの高いサインイン | サインインリスクが高い場合にパスワードリセットを要求 |
| 準拠デバイスの要求 | Intune で管理・準拠しているデバイスのみアクセスを許可 |
| 外部ユーザーのアクセス制限 | ゲストユーザーに追加の条件を設定 |

#### ゼロトラスト戦略との整合

条件付きアクセスポリシーがゼロトラスト戦略と整合しているかを評価するには、以下を確認します。

- すべてのユーザー（管理者を含む）に MFA が適用されているか
- デバイスコンプライアンスが評価されているか
- 継続的なアクセス評価 (CAE) が有効になっているか
- 名前付きロケーションが適切に定義されているか
- レポート専用モードで影響を事前検証しているか

---

### 2-3. 外部コラボレーションのセキュリティ

**Microsoft Entra External ID** は、組織外のユーザー（パートナー、顧客など）との安全なコラボレーションを実現します。

| 機能 | 説明 |
|---|---|
| **B2B コラボレーション** | 外部パートナーが自分の組織の ID を使って自社リソースにアクセス |
| **B2B ダイレクト接続** | Teams の共有チャネルなどのアウトバウンド連携 |
| **外部テナント (B2C)** | カスタマー向けアプリに独自の ID フローを提供 |
| **クロステナント同期** | 複数テナント間でのユーザー・グループの同期 |

**外部 ID のセキュリティベストプラクティス:**

- ゲストユーザーの招待ポリシーを制限する
- ゲストアクセスの有効期限と定期レビューを設定する
- ゲストがアクセスできるリソースを最小限に制限する
- クロステナントアクセス設定で信頼する MFA・デバイスを定義する

---

### 2-4. Microsoft Entra ID の評価とガバナンス

#### Entra ID Protection

Entra ID Protection は、機械学習を活用してリスクの高い ID アクティビティを自動検出・対応します。

**検出されるリスク（例）:**

- 侵害された資格情報の使用（ダークウェブ上の漏洩情報との照合）
- 異常な場所からのサインイン
- マルウェアに感染したデバイスからのアクセス
- 匿名 IP アドレス（Tor など）からのアクセス

**リスクポリシーの設計:**
- **ユーザーリスクポリシー**: 侵害されたと判断されたユーザーにパスワードリセットを強制
- **サインインリスクポリシー**: リスクの高いサインインに MFA を追加要求

#### Entra ID ガバナンス

| 機能 | 説明 |
|---|---|
| **エンタイトルメント管理** | アプリ・グループ・SharePoint へのアクセスパッケージを一元管理 |
| **アクセスレビュー** | ユーザーやゲストのアクセス権を定期的に棚卸し・削除 |
| **ライフサイクルワークフロー** | 入社・異動・退職時の ID プロビジョニングを自動化 |
| **Entra Permissions Management** | マルチクラウド環境の過剰権限を検出・修正（CIEM） |

## モジュール 3: 特権アクセスをセキュリティで保護するソリューションを設計する

### 学習目標

- ゼロトラスト原則とエンタープライズアクセスモデルを使用して特権アクセスを保護できる
- Microsoft Entra ID ガバナンス機能を活用した特権アクセス管理を設計できる
- AI ワークロードやマルチクラウド環境の特権アクセス保護を設計できる

---

### 3-1. エンタープライズアクセスモデル

Microsoft のエンタープライズアクセスモデルは、特権アクセスを段階的に分離・保護するためのフレームワークです。旧来の「Tier モデル」を発展させたものです。

#### 3 つのプレーン

| プレーン | 対象 | 例 |
|---|---|---|
| **コントロールプレーン** | ID システム自体を管理 | Entra ID テナント管理、AD DS ドメインコントローラー |
| **管理プレーン** | リソースのライフサイクルを管理 | Azure サブスクリプション管理、サーバー管理 |
| **データ/ワークロードプレーン** | データやアプリケーションへのアクセス | アプリへのサインイン、ファイルへのアクセス |

コントロールプレーンが侵害されると、他のすべてのプレーンが侵害されるため、最も厳重に保護する必要があります。

---

### 3-2. Microsoft Entra Privileged Identity Management (PIM)

PIM は、特権ロールをジャストインタイムで付与・管理するためのサービスです。

**PIM の主な機能:**

| 機能 | 説明 |
|---|---|
| **適格な割り当て** | 常時ではなく「適格」として割り当て、必要時のみアクティブ化 |
| **時間制限付きアクティベーション** | ロールを有効化できる時間を制限（例: 最大 8 時間） |
| **承認ワークフロー** | ロールのアクティブ化に承認者の許可を必要とする |
| **MFA の強制** | アクティベーション時に MFA を要求 |
| **理由の記録** | アクティベーション時にビジネス上の理由を記録 |
| **通知とアラート** | ロールのアクティブ化をリアルタイムで通知 |
| **アクセスレビュー** | 定期的にロールの割り当てを見直す |
| **監査ログ** | すべての特権アクティビティを記録 |

**PIM の対象:**
- Microsoft Entra ロール（グローバル管理者、セキュリティ管理者など）
- Azure RBAC ロール（所有者、共同作成者など）
- グループメンバーシップ

---

### 3-3. 特権アクセスワークステーション (PAW)

PAW は、管理タスク専用の高度にセキュリティが強化されたデバイスです。

**PAW の設計原則:**

1. **物理的または論理的な分離**: 通常業務（メール、ブラウジング）と管理作業を完全に分離
2. **クリーンソース原則**: 管理ツール・デバイス・ネットワークが管理対象リソースと同等以上のセキュリティレベルを持つ
3. **アプリケーションの制限**: 許可されたアプリケーションのみ実行可能（AppLocker/WDAC）
4. **インターネットアクセスの制限**: 必要なサービスへのアクセスのみ許可

**PAW のプロファイル（展開オプション）:**

- **ハードウェア PAW**: 専用の物理デバイス（最高レベルのセキュリティ）
- **仮想 PAW**: 専用仮想マシン（物理デバイス上に隔離）
- **Azure Virtual Desktop PAW**: クラウド上の仮想デスクトップ

---

### 3-4. 特権アクセスのセキュリティ戦略

#### クラウドテナント管理のセキュリティ

| 対策 | 内容 |
|---|---|
| **グローバル管理者の最小化** | グローバル管理者を最大 5 名以内に制限 |
| **緊急アクセスアカウント** | MFA デバイス喪失など緊急時のブレークグラスアカウント（2 アカウント）を設定 |
| **管理者ロールの棚卸し** | 不要な管理者ロールを定期的に削除 |
| **専用管理アカウント** | 管理作業と通常業務で異なるアカウントを使用 |
| **パスワードレス認証** | FIDO2 セキュリティキーなどによる強固な認証 |

#### クラウドインフラストラクチャエンタイトルメント管理 (CIEM)

**Microsoft Entra Permissions Management** は、マルチクラウド環境（Azure、AWS、GCP）全体の権限を一元的に可視化・管理する CIEM ソリューションです。

- 過剰な権限を持つ ID の検出（未使用権限の特定）
- **Permissions Creep Index (PCI)**: 権限過剰のリスクスコア
- 最小権限への自動修正の推奨
- 継続的なアクセスアクティビティの監視

#### AI ワークロードの特権アクセス

AI システムとエージェントに関する特権アクセスの考慮事項:

- AI サービスに付与する権限を最小化する
- AI エージェントの ID はマネージド ID を使用する
- AI モデルへのアクセスを制御・監査する
- AI によるデータアクセスのスコープを制限する

## モジュール 4: セキュリティオペレーションのソリューションを設計する

### 学習目標

- ログ記録・監査のソリューションを設計できる
- SIEM（セキュリティ情報・イベント管理）のソリューションを設計できる
- SOAR（セキュリティオーケストレーション・自動化・対応）のソリューションを設計できる
- セキュリティワークフローを設計・評価できる
- MITRE ATT&CK フレームワークを活用した脅威検出カバレッジを設計できる

---

### 4-1. 検出と対応のソリューション設計

#### XDR (Extended Detection and Response)

XDR は、複数のセキュリティ製品からのデータを統合し、脅威の検出・調査・対応を自動化するプラットフォームです。

**Microsoft Defender XDR の統合範囲:**

| コンポーネント | 保護対象 |
|---|---|
| **Defender for Endpoint** | Windows/macOS/Linux/iOS/Android デバイス |
| **Defender for Office 365** | Exchange Online、Teams、SharePoint のメール・コラボレーション |
| **Defender for Identity** | オンプレミス AD DS と Entra ID の ID |
| **Defender for Cloud Apps** | SaaS アプリケーション（Shadow IT 含む） |
| **Defender for Cloud** | Azure・AWS・GCP のクラウドワークロード |

#### SIEM (Security Information and Event Management)

**Microsoft Sentinel** は、クラウドネイティブの SIEM + SOAR ソリューションです。

**Sentinel のコアコンポーネント:**

1. **データコネクタ**: Microsoft 製品、サードパーティ製品、カスタムソースからログを収集
2. **分析ルール**: パターンマッチングや機械学習によるアラートの生成
3. **ワークブック**: 可視化ダッシュボードとレポート
4. **インシデント**: アラートをグループ化したインシデント管理
5. **ハンティング**: 脅威を積極的に検索するプロアクティブな調査
6. **UEBA**: ユーザー・エンティティ行動分析
7. **脅威インテリジェンス**: 既知の悪意ある IP やドメインとの照合

### 4-2. 集中ログ記録と監査の設計

#### ログアーキテクチャの設計

**ログの収集・保存戦略:**

```
┌────────────────────────────────────────────┐
│  ログソース                                  │
│  (Azure/M365/オンプレミス/SaaS/カスタム)    │
└──────────────────────┬─────────────────────┘
                       ↓
┌────────────────────────────────────────────┐
│  Microsoft Sentinel (Log Analytics)        │
│  ホットデータ: 即時クエリ (90日〜2年)       │
└──────────────────────┬─────────────────────┘
                       ↓
┌────────────────────────────────────────────┐
│  長期保存ストレージ                          │
│  (Azure Storage / Auxiliary Logs)          │
```

**Purview Audit を活用した集中監査:**

- Microsoft 365 サービス（Exchange、SharePoint、Teams、Entra ID）の操作ログ
- 監査ログの保持ポリシー（標準: 90 日、Audit Premium: 最大 10 年）
- 法的ホールドと電子証拠開示 (eDiscovery) との統合

### 4-3. SOAR とセキュリティワークフロー

**SOAR (Security Orchestration, Automated Response)** は、セキュリティ運用の反復的タスクを自動化し、対応時間を短縮します。

#### Microsoft Sentinel のプレイブック（自動化）

Sentinel のプレイブックは Azure Logic Apps をベースとした自動化ワークフローです。

**自動化の例:**

| トリガー | 自動アクション |
|---|---|
| 疑わしいサインインのアラート | ユーザーアカウントの一時無効化 → 管理者への通知 |
| マルウェア検出 | デバイスの隔離 → チケットの作成 → セキュリティチームへの通知 |
| 大量のデータ転送 | ネットワーク接続のブロック → インシデントの自動作成 |
| 権限昇格の試み | PIM ロールの無効化 → 調査の開始 |

### 4-4. MITRE ATT&CK フレームワークによる脅威検出カバレッジ

MITRE ATT&CK は、実際の攻撃者の戦術・技術・手順 (TTP) を体系化したナレッジベースです。

**ATT&CK の主要な戦術（Tactics）:**

| 戦術 | 説明 |
|---|---|
| **偵察 (Reconnaissance)** | 標的の情報収集 |
| **リソース開発 (Resource Development)** | 攻撃インフラの準備 |
| **初期アクセス (Initial Access)** | ネットワークへの侵入 |
| **実行 (Execution)** | 悪意あるコードの実行 |
| **永続化 (Persistence)** | アクセスの維持 |
| **権限昇格 (Privilege Escalation)** | より高い権限の取得 |
| **防御回避 (Defense Evasion)** | セキュリティ製品の回避 |
| **資格情報アクセス (Credential Access)** | アカウント情報の窃取 |
| **横展開 (Lateral Movement)** | 環境内の移動 |
| **流出 (Exfiltration)** | データの外部への送信 |
| **影響 (Impact)** | システムへの損害・妨害 |

**検出カバレッジの評価:**

Microsoft Sentinel のコンテンツハブには、MITRE ATT&CK にマッピングされた分析ルールが多数含まれています。定期的に以下を評価します。

- カバーされている戦術・技術の割合
- 検出漏れ（ギャップ）の特定
- 新しい検出ルールの追加優先順位付け

### 4-5. セキュリティワークフローの設計

**インシデント対応ワークフロー:**

```
検出 (Detection)
     ↓
トリアージ (Triage) — 優先度付け・影響評価
     ↓
封じ込め (Containment) — 被害の拡大防止
     ↓
根絶 (Eradication) — 攻撃者の排除・マルウェアの除去
     ↓
回復 (Recovery) — サービスの復旧
     ↓
事後分析 (Post-Incident Analysis) — 再発防止策の策定
```

**脅威ハンティング:**

脅威ハンティングは、アラートが発生していない状態でも積極的に脅威を探索するプロアクティブなアプローチです。

- **仮説主導**: ATT&CK TTP に基づいた仮説を立ててデータを調査
- **KQL (Kusto Query Language)**: Sentinel の強力なクエリ言語を活用
- **ハンティングクエリの管理**: 有効なハンティングクエリを分析ルール化

## まとめ: ラーニングパス全体の重要ポイント

### 設計領域と主要ソリューションの対応

| 設計領域 | 主要な Microsoft ソリューション |
|---|---|
| 規制コンプライアンス | Purview Compliance Manager、Defender for Cloud、Azure Policy |
| ID・アクセス管理 | Microsoft Entra ID、条件付きアクセス、Entra ID Protection |
| 特権アクセス | PIM、PAW、Permissions Management |
| セキュリティオペレーション | Microsoft Sentinel、Defender XDR、Purview Audit |

### 重要な設計原則

1. **ID ファースト**: ID をセキュリティの新しい境界として設計する
2. **最小権限の徹底**: PIM によるジャストインタイムアクセスを活用する
3. **統合された SecOps**: Sentinel と Defender XDR を連携させた包括的な検出・対応体制を構築する
4. **コンプライアンスの自動化**: Azure Policy と Defender for Cloud で継続的なコンプライアンス評価を自動化する
5. **MITRE ATT&CK への対応**: 脅威検出カバレッジをフレームワークに基づいて系統的に評価・改善する
