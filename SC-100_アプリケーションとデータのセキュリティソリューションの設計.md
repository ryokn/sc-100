### 1-1. Microsoft 365 セキュリティの評価

Microsoft 365 は多くの組織の中核的な生産性プラットフォームであり、メール・コラボレーション・データの保護が不可欠です。

#### Microsoft セキュアスコアによる評価

Microsoft セキュアスコアは、Microsoft 365 と Azure の現在のセキュリティ態勢を定量的に測定するツールです。

**セキュアスコアの構成要素:**

| カテゴリ | 評価対象 |
|---|---|
| **ID** | MFA の有効化、レガシー認証のブロック、特権アカウントの保護 |
| **デバイス** | Intune 登録率、コンプライアンスポリシーの適用、エンドポイント保護 |
| **アプリ** | OAuth アプリの制御、クラウドアプリのリスク評価 |
| **データ** | 機密情報の保護、DLP ポリシーの設定 |
| **インフラ** | Azure サービスのセキュリティ設定 |

**スコア向上のアプローチ:**
1. 現在のスコアと推奨事項を確認する
2. 高スコア・低コストの改善アクションを優先する
3. 実装後のスコア変化を追跡する
4. 業界平均や類似組織との比較で自組織の位置を把握する

### 1-2. Microsoft 365 の主要セキュリティ機能

#### Microsoft Defender for Office 365

メール・コラボレーションプラットフォームを保護するための脅威対策ソリューションです。

**保護レイヤー:**

| 機能 | 説明 |
|---|---|
| **Exchange Online Protection (EOP)** | スパム、マルウェア、フィッシングの基本的なフィルタリング |
| **安全なリンク (Safe Links)** | URL をリアルタイムでスキャンし、悪意あるリンクをクリック時にブロック |
| **安全な添付ファイル (Safe Attachments)** | 添付ファイルを仮想環境で実行して悪意あるコードを検出 |
| **フィッシング対策** | なりすましや偽装攻撃の検出 |
| **キャンペーンビュー** | 組織に対する大規模なフィッシングキャンペーンの可視化 |
| **脅威トラッカー** | 最新の脅威動向の追跡 |
| **攻撃シミュレーション** | フィッシングシミュレーションによるユーザー教育 |

#### Microsoft Purview を使用したデータセキュリティ評価

Microsoft Purview は、Microsoft 365 のデータを保護・管理するための包括的なコンプライアンスプラットフォームです。

**Microsoft 365 のデータセキュリティに関する主要機能:**

| 機能 | 説明 |
|---|---|
| **情報保護 (Information Protection)** | データの機密度に基づいてラベル付け・暗号化・アクセス制御 |
| **データ損失防止 (DLP)** | 機密情報の意図しない共有・漏洩を防止 |
| **インサイダーリスク管理** | 内部からの意図的・意図しないリスクを検出 |
| **コミュニケーションコンプライアンス** | 通信内容のポリシー違反を監視 |
| **電子証拠開示 (eDiscovery)** | 法的調査のためのデータ検索・収集・エクスポート |
| **レコード管理** | 規制要件に基づくコンテンツの保持・廃棄管理 |

### 1-3. Microsoft Copilot for Microsoft 365 のデータセキュリティ

AI アシスタントである Microsoft Copilot for Microsoft 365 を組織に導入する際は、データセキュリティとコンプライアンスの観点からの評価が重要です。

#### Copilot のデータアクセスモデル

Copilot は、ユーザーがアクセス権限を持つデータのみにアクセスします（Microsoft Graph 経由）。つまり：

- ユーザーが閲覧できないファイルには Copilot もアクセスできない
- 過剰共有されたデータは Copilot を通じて情報漏洩のリスクになる

#### Copilot 導入前のセキュリティ評価チェックリスト

- [ ] 過剰共有コンテンツの特定と修正（SharePoint の「組織内の全員」共有の見直し）
- [ ] 秘密度ラベルの適用状況の確認
- [ ] 機密データへのアクセス権限の棚卸し
- [ ] Microsoft Purview のデータ分類とコンプライアンスポリシーの設定
- [ ] **Purview AI Hub** を使用した Copilot のデータアクセスと使用状況の監視

#### Purview AI Hub

Purview AI Hub は、組織内の AI ツール（Copilot を含む）の使用状況を可視化・管理するためのダッシュボードです。

- AI ツールへのデータ入力とアウトプットの監視
- 機密情報が AI に渡されていないかの確認
- AI 使用に関するコンプライアンスポリシーの適用

### 2-1. 既存アプリケーションポートフォリオのセキュリティ評価

#### アプリケーションセキュリティ態勢の評価

組織のアプリケーションポートフォリオを評価する際の主要な観点:

1. **インベントリの作成**: すべてのアプリケーション（SaaS・PaaS・カスタム開発）のリストアップ
2. **リスク分類**: ビジネスの重要度・処理するデータの機密性・外部公開状況に基づく分類
3. **セキュリティコントロールの評価**: 認証方式・認可モデル・暗号化・ログの確認
4. **脆弱性の特定**: 既知の脆弱性・設定ミス・アップデート状況の確認

#### Microsoft Defender for Cloud Apps を使用した評価

Defender for Cloud Apps は、クラウドアプリのリスクを評価・管理する CASB (Cloud Access Security Broker) です。

| 機能 | 説明 |
|---|---|
| **Cloud Discovery** | 組織で使用されているクラウドアプリを自動検出（Shadow IT の可視化） |
| **アプリカタログ** | 25,000 以上のアプリのリスクスコアと詳細情報 |
| **アプリコネクタ** | 主要 SaaS アプリ（Salesforce、Box、GitHub など）と API 連携 |
| **セッションコントロール** | リアルタイムでのアプリアクセスのモニタリングと制御 |
| **異常検出** | 通常とは異なる使用パターンの検出 |

### 2-2. 脅威モデリング

脅威モデリングは、アプリケーションの設計段階から潜在的なセキュリティ脅威を特定・対処するための体系的なアプローチです。

#### STRIDE モデル

Microsoft が開発した代表的な脅威分類フレームワーク:

| 脅威カテゴリ | 英語 | 説明 | 対策例 |
|---|---|---|---|
| **なりすまし** | Spoofing | 別の ID になりすます | 強力な認証の実装 |
| **改ざん** | Tampering | データやコードを不正に変更する | データ整合性の検証・署名 |
| **否認** | Repudiation | 実行した操作を否定する | 監査ログの取得 |
| **情報漏洩** | Information Disclosure | 意図しない情報の公開 | 暗号化・アクセス制御 |
| **サービス拒否** | Denial of Service | サービスの可用性を損なう | レート制限・スケーリング |
| **権限昇格** | Elevation of Privilege | より高い権限を不正取得する | 最小権限・入力検証 |

#### 脅威モデリングの実施手順

```
1. スコープの定義
   （アプリケーションの範囲・資産の特定）
         ↓
2. データフロー図の作成
   （信頼境界・データの流れを可視化）
         ↓
3. 脅威の特定
   （STRIDE などを使って各コンポーネントへの脅威をリストアップ）
         ↓
4. 脅威の評価
   （リスクスコアによる優先順位付け）
         ↓
5. 対策の設計
   （脅威ごとのセキュリティコントロールを設計）
         ↓
6. 検証
   （対策が脅威に対して有効かを確認）
```

### 2-3. アプリケーションセキュリティのフルライフサイクル戦略

#### Microsoft Security Development Lifecycle (SDL)

SDL は、開発プロセス全体にセキュリティを組み込むためのプロセスです。

**フェーズ別のセキュリティ活動:**

| フェーズ | 主な活動 |
|---|---|
| **要件定義** | セキュリティ要件の定義、プライバシーリスクの評価 |
| **設計** | 脅威モデリング、セキュアな設計パターンの適用 |
| **実装** | セキュアコーディング標準、静的コード解析 (SAST) |
| **テスト** | セキュリティテスト、動的解析 (DAST)、ペネトレーションテスト |
| **デプロイ** | セキュアな構成、シークレット管理 |
| **運用** | 継続的な監視、脆弱性管理、インシデント対応 |

#### CI/CD パイプラインのセキュリティ

DevSecOps の実践として、CI/CD パイプラインに以下のセキュリティチェックを組み込みます。

| チェック | 目的 | ツール例 |
|---|---|---|
| **SAST** (静的コード解析) | コード内の脆弱性を検出 | CodeQL, SonarQube |
| **SCA** (ソフトウェア構成分析) | オープンソース依存関係の脆弱性 | Dependabot, Snyk |
| **シークレットスキャン** | コード内のシークレット（API キーなど）を検出 | GitHub Secret Scanning |
| **コンテナスキャン** | コンテナイメージの脆弱性 | Trivy, Defender for Containers |
| **IaC スキャン** | Infrastructure as Code のセキュリティ設定ミス | Defender for DevOps |

**GitHub Advanced Security と Microsoft Defender for DevOps:**

Microsoft Defender for DevOps は、GitHub や Azure DevOps パイプライン全体のセキュリティ態勢を一元的に可視化・管理します。

### 2-4. ワークロード ID のセキュリティ設計

ワークロード ID は、アプリケーション・サービス・仮想マシン・コンテナなどの非人間 ID です。

#### マネージド ID

Azure のマネージド ID は、Azure リソースに対して Entra ID の ID を自動的に付与します。シークレット（パスワード・証明書）の管理が不要になるため、資格情報の漏洩リスクを排除できます。

| 種類 | 説明 | 適用場面 |
|---|---|---|
| **システム割り当て** | リソースのライフサイクルに紐付け（リソース削除で自動削除） | 単一リソース専用の ID が必要な場合 |
| **ユーザー割り当て** | 独立したリソースとして管理（複数リソースで共有可能） | 複数リソースで同じ ID を共有する場合 |

#### アプリケーション ID のセキュリティベストプラクティス

- **マネージド ID の優先使用**: シークレットを使う代わりにマネージド ID を優先
- **最小権限の適用**: アプリに必要な権限のみを割り当てる
- **シークレットの安全な管理**: シークレットが必要な場合は Azure Key Vault を使用
- **証明書の定期的な更新**: シークレットや証明書の有効期限を監視・自動更新
- **ワークロード ID フェデレーション**: GitHub Actions など外部 IdP との連携（シークレット不要）

### 2-5. API のセキュリティ設計

現代のアプリケーションは API を通じて通信することが多く、API のセキュリティは重要な設計要素です。

**API セキュリティのベストプラクティス:**

| 項目 | ベストプラクティス |
|---|---|
| **認証・認可** | OAuth 2.0 / OpenID Connect を使用、JWT トークンを適切に検証 |
| **API ゲートウェイ** | Azure API Management で一元的にポリシーを適用 |
| **レート制限** | API の過剰使用や DDoS 攻撃を防ぐスロットリング |
| **入力検証** | すべての入力データを検証してインジェクション攻撃を防止 |
| **TLS の強制** | すべての API 通信を HTTPS に限定 |
| **機密情報の保護** | レスポンスに不要な機密情報を含めない |
| **API バージョン管理** | 旧バージョンの適切な廃止とセキュリティパッチの適用 |

### 3-1. Microsoft Purview によるデータ保護

#### データ保護のライフサイクル

```
データの把握 (Know your data)
  → 機密情報の自動検出と分類
         ↓
データの保護 (Protect your data)
  → 秘密度ラベルの適用・暗号化・アクセス制御
         ↓
データ損失の防止 (Prevent data loss)
  → DLP ポリシーによる不適切な共有の防止
         ↓
ガバナンス (Govern your data)
  → 保持・廃棄・コンプライアンス要件の充足
```

#### 情報保護（秘密度ラベル）

秘密度ラベルは、データの機密レベルを分類し、適切な保護を自動的に適用するメカニズムです。

**一般的なラベル階層の例:**

| ラベル | 説明 | 保護の例 |
|---|---|---|
| **公開 (Public)** | 誰でも閲覧可能な情報 | 保護なし |
| **社内限定 (General)** | 組織内で共有する一般的な業務情報 | 組織内の共有を許可 |
| **機密 (Confidential)** | 制限された関係者のみ閲覧可能 | 暗号化・特定グループのみアクセス可 |
| **極秘 (Highly Confidential)** | 厳重な管理が必要な最高機密情報 | 暗号化・承認されたユーザーのみ |

**ラベルで適用できる保護:**

- 暗号化（Azure Rights Management）
- アクセス制御（特定ユーザー・グループのみ）
- コンテンツマーキング（ヘッダー、フッター、透かし）
- 条件付きアクセスポリシーとの連携

#### データ損失防止 (DLP)

DLP ポリシーは、機密情報が意図せず組織外に漏洩することを防ぐためのポリシーです。

**DLP ポリシーの対象:**

- Exchange Online（メール）
- SharePoint Online / OneDrive
- Teams のチャットとチャネル
- エンドポイントデバイス
- Microsoft 365 Copilot

**DLP の動作例:**

```
メールに日本のマイナンバーが含まれている場合
     ↓
DLP エンジンが検知
     ↓
設定に応じて:
  - 送信をブロックする
  - ポリシーヒントを表示して警告する
  - 上司への CC を自動追加する
  - インシデントレポートを生成する
```

### 3-2. Azure ワークロードのデータセキュリティ

#### Azure SQL のデータセキュリティ

| 機能 | 説明 |

#### Azure Storage のデータセキュリティ

| 機能 | 説明 |

| **Microsoft Defender for Storage** | マルウェアスキャン、機密情報検出、異常アクセス検出 |

#### Azure Cosmos DB・Synapse Analytics のデータセキュリティ

| サービス | 主なセキュリティ機能 |
|---|---|
| **Azure Cosmos DB** | 保存時・転送時の暗号化、ファイアウォール、プライベートエンドポイント、Microsoft Defender for Cosmos DB |
| **Azure Synapse Analytics** | 列レベル暗号化、行レベルセキュリティ、動的データマスキング、Microsoft Defender for SQL |

### 3-3. データの暗号化ソリューション

#### 暗号化の 2 つの状態

| 状態 | 英語 | 説明 | 主な技術 |
|---|---|---|---|
| **保存データ** | Data at Rest | ストレージに保存された静的なデータ | AES 暗号化、TDE |
| **転送中データ** | Data in Transit | ネットワークで転送されているデータ | TLS/HTTPS |
| **使用中データ** | Data in Use | メモリ上で処理中のデータ | Confidential Computing |

#### Azure Key Vault による鍵管理

Azure Key Vault は、暗号化キー・シークレット・証明書を安全に管理するためのサービスです。

**3 つの主要機能:**

1. **キーコンテナー**: ソフトウェア保護またはHSM保護の暗号化キーを管理
2. **シークレット管理**: 接続文字列・パスワード・API キーなどを安全に保存
3. **証明書管理**: SSL/TLS 証明書のライフサイクルを自動管理

**Key Vault のベストプラクティス:**

- マネージド ID を使用してアプリから Key Vault にアクセスする
- アクセスポリシーではなく Azure RBAC を使用して権限を管理する
- 論理的な削除（Soft Delete）と消去保護を有効にする
- Key Vault のアクセスログを監視する
- 重要なキーはユーザー管理HSM (Managed HSM) を使用する

#### カスタマーマネージドキー (CMK) vs プラットフォームマネージドキー (PMK)

| 比較項目 | PMK（Microsoft 管理） | CMK（お客様管理） |
|---|---|---|
| **管理の複雑さ** | 低い（自動管理） | 高い（自社でキー管理） |
| **コスト** | 追加コストなし | Key Vault のコストが発生 |
| **コントロール** | Microsoft に依存 | 完全なコントロール |
| **適用場面** | 標準的なデータ保護 | 規制要件・高機密データ |

## モジュール 4: 対話型ケーススタディ — アプリとデータのセキュリティ保護

### シナリオの概要

実際のビジネスシナリオに対して、アプリケーションとデータのセキュリティを保護するソリューションを設計します。

### 評価される設計スキル

**アプリケーションセキュリティの観点:**
- 既存アプリケーションのセキュリティリスクの評価と優先度付け
- セキュアな認証・認可の設計
- API セキュリティの強化
- DevSecOps プロセスへの移行計画

**データセキュリティの観点:**
- データ分類と秘密度ラベルの設計
- DLP ポリシーの設計
- Azure データサービスの暗号化設定
- データアクセスの監視と異常検出

**Microsoft 365 セキュリティの観点:**
- Defender for Office 365 の設定最適化
- Purview コンプライアンスポリシーの設計
- Copilot 導入に向けたセキュリティ評価

## まとめ: ラーニングパス全体の重要ポイント

### 設計領域と主要ソリューションの対応

| 設計領域 | 主要な Microsoft ソリューション |
|---|---|
| Microsoft 365 セキュリティ | Defender for Office 365、Microsoft セキュアスコア、Purview |
| アプリケーション保護 | Defender for Cloud Apps、Defender for DevOps、API Management |
| データ保護 | Purview 情報保護、Purview DLP、Key Vault |
| Azure データサービス | Defender for SQL、Defender for Storage、TDE、CMK |

### 重要な設計原則

1. **データを知ること (Know Your Data)**: 機密データがどこにあるかを把握し、分類・ラベル付けを行う
2. **ゼロトラストをアプリに適用する**: アプリやサービスにも最小権限と継続的な検証を適用する
3. **「左シフト」によるセキュリティ**: 開発の早い段階でセキュリティを組み込む（SDL・脅威モデリング）
4. **シークレットをコードから排除する**: マネージド ID と Key Vault を活用してシークレットをコードに含めない
5. **多層防御のデータ保護**: 暗号化・アクセス制御・DLP・監視を組み合わせたデータ保護を設計する

### 試験での重要なポイント

- Microsoft Purview の各コンポーネント（情報保護、DLP、Compliance Manager、Audit）の役割を明確に区別する
- Azure データサービス（SQL、Storage、Cosmos DB）それぞれのセキュリティ機能を把握する
- マネージド ID と Key Vault の組み合わせがシークレット管理のベストプラクティスであることを覚える
- Copilot の導入には Purview によるデータ分類と過剰共有の修正が前提となることを理解する
