### 1-1. セキュリティのベストプラクティスとは何か

**ベストプラクティス**とは、経験と実績から導き出された、特定の目標を効果的・効率的に達成するための方法論や手順のことです。サイバーセキュリティにおけるベストプラクティスは、攻撃から組織を守るための実証済みのアプローチを提供します。

#### アンチパターン（避けるべき設計・運用）

セキュリティ設計では、以下のような「アンチパターン」を避けることが重要です。

| アンチパターン | 問題点 | 推奨されるアプローチ |
|---|---|---|
| セキュリティの後付け | 開発完了後にセキュリティを追加しようとする | 設計段階からセキュリティを組み込む (Security by Design) |
| 境界防御への過度な依存 | ファイアウォールの内側は安全と仮定する | ゼロトラストを採用し、内部・外部を問わず検証する |
| 最小権限の無視 | ユーザーに必要以上の権限を付与する | 最小権限の原則 (Least Privilege) を徹底する |
| パッチ管理の怠慢 | 既知の脆弱性があってもアップデートを先延ばしにする | 継続的な脆弱性管理と迅速なパッチ適用 |
| セキュリティの孤立化 | セキュリティチームが他部門から孤立して活動する | 開発・運用・セキュリティが連携する DevSecOps の実践 |

#### サイバーセキュリティアーキテクトの役割

サイバーセキュリティアーキテクトは、組織のサイバーセキュリティ戦略を、資産・ビジネス・オペレーションを保護する能力へと変換する役割を担います。具体的には以下を行います。

- ゼロトラスト原則とベストプラクティスに従ったセキュリティソリューションの設計・実装・維持
- ID、デバイス、データ、AI、アプリケーション、ネットワーク、インフラ、DevOps にわたるセキュリティ戦略の策定
- ガバナンス・リスク・コンプライアンス (GRC)、セキュリティオペレーション、セキュリティ態勢管理のためのソリューション設計
- 組織全体のリーダーやセキュリティ担当者と継続的に協力し、ビジネスニーズに合ったサイバーセキュリティ戦略を計画・実装

### 1-2. ゼロトラストの概念と基本原則

#### ゼロトラストとは

ゼロトラストは「境界内のすべてを信頼する」という従来のネットワーク境界モデルを根本から見直したセキュリティモデルです。**「決して信頼せず、常に検証する (Never Trust, Always Verify)」** という原則のもと、あらゆるアクセスリクエストを検証対象として扱います。

ゼロトラストが必要とされる背景:

- クラウドサービスの普及により、社内ネットワークの境界が曖昧になった
- リモートワークの拡大で、従業員が様々な場所・デバイスからアクセスするようになった
- 高度化する内部脅威や、信頼された内部ネットワークを経由した攻撃が増加している

#### ゼロトラストの 3 つの基本原則

**1. 明示的に確認する (Verify Explicitly)**

利用可能なすべてのデータポイントに基づいて、常に認証・認可を行います。確認すべきデータポイントには以下が含まれます。

- ユーザーの ID（本人確認）
- アクセス元の場所（地理的位置情報）
- デバイスの正常性（コンプライアンス状態）
- アクセス対象のサービスまたはワークロード
- データの機密度分類
- 異常なアクセスパターンや振る舞い

**2. 最小権限アクセスを使用する (Use Least-Privilege Access)**

ジャストインタイム (JIT) およびジャストイナフアクセス (JEA)、リスクベースの適応型ポリシー、データ保護を活用して、ユーザーのアクセスを必要最小限に制限します。

- **JIT (Just-In-Time)**: 必要なときだけ、必要な期間だけ権限を付与する
- **JEA (Just-Enough-Access)**: 特定のタスクに必要な最小限の権限のみを付与する
- リスクベースの条件付きアクセスポリシーによる動的なアクセス制御

**3. 侵害を想定する (Assume Breach)**

侵害はすでに発生している、あるいは必ず発生するという前提で設計・運用します。

- 被害範囲を最小化するためのネットワークセグメンテーション
- エンドツーエンドの暗号化の使用
- 分析・可視性の向上による脅威検出の推進
- 継続的な監視とログの収集

#### ゼロトラスト導入フレームワーク

Microsoft は、組織がゼロトラストを段階的に導入するための**ゼロトラスト採用フレームワーク**を提供しています。これは以下のビジネスシナリオで構成されています。

1. **リモートワークと現場作業者のセキュリティ保護** — 場所を問わない安全なアクセスの実現
2. **重要なビジネスアプリとデータの保護** — 重要資産への厳格なアクセス制御
3. **最新のセキュリティオペレーションの実現** — 脅威の検出・対応の高度化
4. **最小権限アクセスの近代化** — 特権アクセス管理の強化
5. **テレメトリとアナリティクスの活用** — データドリブンなセキュリティ意思決定

また、**ゼロトラスト迅速近代化計画 (Zero Trust Rapid Modernization Plan / RaMP)** は、優先度の高い取り組みに集中して迅速にゼロトラストの成熟度を高めるためのロードマップです。

### 1-3. 主要なベストプラクティスフレームワーク

#### フレームワークの全体像と相互関係

Microsoft のセキュリティフレームワークは相互に補完し合いながら、包括的なセキュリティ戦略の構築を支援します。

```
┌─────────────────────────────────────────────────────┐
│              ビジネス目標・セキュリティ戦略              │
└──────────────────────┬──────────────────────────────┘
                       │
         ┌─────────────┴─────────────┐
         ▼                           ▼
┌─────────────────┐       ┌─────────────────────┐
│  CAF            │       │  WAF                │
│  (クラウド導入   │       │  (Well-Architected  │
│   フレームワーク)│       │   フレームワーク)    │
└────────┬────────┘       └──────────┬──────────┘
         │                           │
         └─────────────┬─────────────┘
                       ▼
           ┌─────────────────────┐
           │  MCRA + MCSB        │
           │  (参照アーキテクチャ  │
           │  + セキュリティ      │
           │  ベンチマーク)       │
           └─────────────────────┘
```

#### CAF (Microsoft Cloud Adoption Framework / クラウド導入フレームワーク)

CAF は、組織がクラウドへの移行を計画・実行するための包括的なガイダンスです。セキュリティに関しては **「セキュア手法 (Secure Methodology)」** として、クラウド導入プロセス全体を通じてセキュリティを統合する方法を提供します。

主なフェーズ:
- **戦略 (Strategy)**: ビジネス目標をセキュリティ要件に変換
- **計画 (Plan)**: セキュリティロードマップの作成
- **準備 (Ready)**: セキュアなランディングゾーンの設計
- **導入 (Adopt)**: セキュアな移行・新規開発
- **ガバナンス (Govern)**: 継続的なポリシーとコンプライアンス管理
- **管理 (Manage)**: セキュリティオペレーションと監視

#### WAF (Microsoft Azure Well-Architected Framework / 優れたアーキテクチャのフレームワーク)

WAF は、クラウドソリューションの設計と品質向上のためのガイダンスフレームワークです。5つの柱で構成され、セキュリティはその一つとして位置づけられています。

| 柱 | 説明 |
|---|---|
| **信頼性 (Reliability)** | 障害への耐性と迅速な回復 |
| **セキュリティ (Security)** | データ・アプリ・インフラの保護 |
| **コスト最適化 (Cost Optimization)** | 価値を最大化しながらコストを削減 |
| **オペレーショナルエクセレンス (Operational Excellence)** | 運用の効率化と自動化 |
| **パフォーマンス効率 (Performance Efficiency)** | スケーラビリティと効率的なリソース利用 |

#### MCRA (Microsoft Cybersecurity Reference Architecture / Microsoft サイバーセキュリティ参照アーキテクチャ)

MCRA は、Microsoft のサイバーセキュリティ能力に関する詳細な技術図を集めた参照アーキテクチャです。以下の領域をカバーしています。

- ゼロトラストユーザーアクセス
- セキュリティオペレーション
- 運用技術 (OT) セキュリティ
- マルチクラウド・クロスプラットフォームの能力
- 攻撃チェーン対策
- Azure ネイティブのセキュリティ制御
- セキュリティ組織機能

#### MCSB (Microsoft Cloud Security Benchmark / Microsoft クラウドセキュリティベンチマーク)

MCSB は、Azure とマルチクラウド環境に適用されるセキュリティのベストプラクティスと推奨事項を提供します。CIS、NIST などの業界標準フレームワークに基づいており、Microsoft Defender for Cloud を通じてコンプライアンス状態を自動評価できます。

主要なコントロール領域:
- ネットワークセキュリティ
- ID 管理
- 特権アクセス
- データ保護
- 資産管理
- ログと脅威検出
- インシデント対応
- 脆弱性管理
- エンドポイントセキュリティ

- Well-Architected フレームワーク (WAF) を理解し、セキュリティを含む堅牢な設計原則に沿ったクラウドソリューションを設計できる

---

### 2-1. クラウド導入フレームワーク (CAF) を活用したセキュリティ設計

#### CAF セキュア手法の概要

CAF のセキュア手法は、クラウド導入ライフサイクル全体を通じてセキュリティを統合するための体系的なアプローチです。セキュリティ要件をビジネス目標に合わせ、技術的な実装へと変換します。

#### ポリシー主導のガバナンス

ポリシー主導のガバナンスにより、リソース全体に一貫したセキュリティを自動的に適用します。

- **Azure Policy**: スケールでセキュリティポリシーを強制する
- **Azure Blueprints**: 準拠した環境を一貫してデプロイする
- **Azure Landing Zones**: セキュアなクラウド環境の基盤を提供する

このアプローチにより、同じ作業を手動で繰り返す無駄を省き、セキュリティの一貫性を確保できます。

#### Azure ランディングゾーンによるセキュリティ実装

Azure ランディングゾーンは、スケーラブルで管理された、セキュアなクラウド環境の基盤です。セキュリティの観点から以下を含みます。

- **ID・アクセス管理**: Microsoft Entra ID を中核とした ID 基盤
- **ネットワークセキュリティ**: ネットワーク境界の定義とセグメンテーション
- **管理とガバナンス**: ポリシーとコンプライアンスの継続的な管理
- **セキュリティ監視**: 集中ログ・監視基盤の構築

#### DevSecOps プロセスの設計

CAF のベストプラクティスに沿った DevSecOps プロセスでは、セキュリティを開発・デプロイのサイクル全体に組み込みます。

従来の「ウォーターフォールモデル」では、セキュリティは開発完了後の品質ゲートとして機能していました。しかし DevSecOps では、セキュリティをプロセスの早い段階から統合します（「左シフト」と呼ばれる考え方）。

**DevSecOps の 3 つの要件タイプ:**

1. **ビジネス要件 (Dev)**: アプリケーションはビジネスとユーザーのニーズを満たさなければならない
2. **セキュリティ要件 (Sec)**: アプリケーションは急速に進化する攻撃に対して回復力を持ち、セキュリティの革新を活用しなければならない
3. **IT 運用要件 (Ops)**: アプリケーションは信頼性が高く、効率的に動作しなければならない

#### クラウドセキュリティ態勢の向上

Microsoft Defender for Cloud を活用して、ハイブリッドおよびマルチクラウド環境全体のセキュリティ態勢を可視化・改善します。

- **セキュアスコア**: 現在のセキュリティ態勢を定量的に評価
- **推奨事項**: 優先度付きの具体的な改善アクションを提供
- **規制コンプライアンス**: MCSB、CIS、PCI DSS などの業界標準への準拠状況を評価

---

### 2-2. Well-Architected フレームワーク (WAF) を活用したセキュリティ設計

#### WAF セキュリティの柱：主要な設計原則

1. **ゼロトラストを前提に計画する**: ゼロトラスト原則に基づいてすべてのコンポーネントを設計する
2. **ID・アクセスをセキュリティの基盤とする**: 強力な ID 管理をセキュリティの中心に据える
3. **最小権限を徹底する**: 過剰な権限付与を防ぎ、不必要なアクセスを制限する
4. **多層防御 (Defense in Depth)**: 単一のセキュリティコントロールに依存しない
5. **ネットワークセグメンテーション**: リソースを分離して横方向の移動 (Lateral Movement) を防ぐ

#### セキュリティ設計のチェックリスト

**ID とアクセス管理:**
- [ ] 多要素認証 (MFA) を有効化する
- [ ] 条件付きアクセスポリシーを構成する
- [ ] 特権 ID 管理 (PIM) を導入する
- [ ] ゲストアクセスのライフサイクルを管理する

**データ保護:**
- [ ] 保存データと転送中データを暗号化する
- [ ] データ分類と情報保護ポリシーを実装する
- [ ] 機密情報へのアクセスを監視する

**ネットワークセキュリティ:**
- [ ] ネットワークセグメンテーションを適用する
- [ ] DDoS 保護を有効化する
- [ ] Web Application Firewall (WAF) を構成する

**監視とインシデント対応:**
- [ ] セキュリティログを集中収集する
- [ ] アラートとインシデント対応プロセスを定義する
- [ ] 定期的にセキュリティ態勢をレビューする

## モジュール 3: MCRA・MCSB に沿ったソリューションの設計

### 学習目標

- Microsoft Cybersecurity Reference Architecture (MCRA) と Microsoft Cloud Security Benchmark (MCSB) を理解し、より安全なソリューションを設計できる
- Security Adoption Framework (SAF) を活用したセキュリティ導入計画を立案できる
- インサイダー脅威、外部攻撃、サプライチェーン侵害、AI 固有のリスクから保護するソリューションを設計できる

---

### 3-1. Microsoft Cybersecurity Reference Architecture (MCRA)

#### MCRA の構成と目的

MCRA は Microsoft のサイバーセキュリティ能力を視覚的に示した包括的な参照アーキテクチャです。セキュリティアーキテクトが統合アーキテクチャを設計する際の「地図」として機能します。

**Microsoft セキュリティ製品の統合全体像:**

| レイヤー | 主な製品・サービス |
|---|---|
| **SIEM / XDR 統合** | Microsoft Sentinel + Microsoft Defender XDR |
| **エンドポイント保護** | Microsoft Defender for Endpoint |
| **メール・コラボレーション保護** | Microsoft Defender for Office 365 |
| **ID 保護** | Microsoft Defender for Identity, Entra ID Protection |
| **クラウドアプリ保護** | Microsoft Defender for Cloud Apps |
| **クラウドセキュリティ態勢** | Microsoft Defender for Cloud (CSPM/CWPP) |
| **脅威インテリジェンス** | Microsoft Defender 脅威インテリジェンス |

#### ゼロトラストユーザーアクセスアーキテクチャ

MCRA が示すゼロトラストユーザーアクセスの核心は、**ID を新しいセキュリティ境界**として位置づけることです。

アクセスフローの概要:

1. ユーザーが任意の場所からアクセスを試みる
2. Microsoft Entra ID がユーザー・デバイス・リスクシグナルを評価
3. 条件付きアクセスポリシーが適用（MFA の要求、デバイスコンプライアンス確認など）
4. アクセスが許可または拒否される
5. 許可された場合も、セッション中の継続的なモニタリングが実施される

#### セキュリティオペレーションアーキテクチャ

効果的なセキュリティオペレーションは以下の要素で構成されます。

| 機能 | Microsoft ソリューション | 説明 |
|---|---|---|
| **SIEM** | Microsoft Sentinel | セキュリティイベントの収集・分析・相関 |
| **XDR** | Microsoft Defender XDR | エンドポイント・メール・ID・クラウドアプリの統合検出と対応 |
| **SOAR** | Sentinel プレイブック | インシデント対応の自動化とオーケストレーション |
| **脅威インテリジェンス** | Defender 脅威インテリジェンス | 最新の脅威情報の収集・活用 |
| **CSPM** | Defender for Cloud | クラウドセキュリティ態勢の継続的な評価と改善 |

MCSB はクラウドワークロードを保護するための包括的なセキュリティコントロールを提供します。

| コントロール領域 | 略称 | 主な内容 |

| ネットワークセキュリティ | NS | ネットワーク境界の保護、通信の暗号化 |

| 特権アクセス | PA | 特権アカウントの管理と保護 |
| データ保護 | DP | 暗号化、データ分類、DLP |
| 資産管理 | AM | リソースのインベントリと脆弱性管理 |
| ログと脅威検出 | LT | ログ収集、アラート、監視 |
| インシデント対応 | IR | インシデント計画と対応プロセス |
| 脆弱性管理 | PV | セキュリティスキャンとパッチ管理 |
| エンドポイントセキュリティ | ES | エンドポイント保護と構成管理 |
| バックアップと回復 | BR | データバックアップと回復性 |
| DevOps セキュリティ | DS | セキュアな開発・デプロイプロセス |

#### 能力とコントロールに関するベストプラクティス

**ID の保護:**
- Microsoft Entra ID を使用した一元的な ID 管理
- 多要素認証 (MFA) の全ユーザーへの適用
- 条件付きアクセスポリシーによるリスクベースのアクセス制御
- 特権 ID 管理 (PIM) による特権アクセスのジャストインタイム提供

**データの保護:**
- Microsoft Purview によるデータ分類と情報保護
- 保存データ・転送中データの暗号化
- データ損失防止 (DLP) ポリシーの実装
- Azure Key Vault による秘密・キー・証明書の管理

**ネットワークの保護:**
- ネットワークセグメンテーションとマイクロセグメンテーション
- Azure Firewall と Web Application Firewall の活用
- Azure DDoS Protection の有効化
- Private Endpoint によるサービスへのプライベートアクセス

### 3-3. 攻撃への保護ベストプラクティス

#### インサイダー脅威への対策

インサイダー脅威（悪意のある内部者や、意図せず被害を引き起こす内部者）に対するベストプラクティス:

- **最小権限の実装**: 職務に必要な最小限のアクセスのみ付与
- **特権アクセスワークステーション (PAW)**: 管理作業専用の安全なデバイスを使用
- **ユーザー行動分析 (UEBA)**: 異常なアクセスパターンを自動検出
- **定期的なアクセスレビュー**: Microsoft Entra ID のアクセスレビュー機能を活用
- **Microsoft Purview Insider Risk Management**: 内部リスクの検出と調査

#### 外部攻撃への対策

外部の攻撃者に対するベストプラクティス:

- **攻撃対象領域の縮小**: 不必要に公開されているサービスの最小化
- **エクスポージャー管理**: Microsoft Defender の外部攻撃対象管理 (EASM)
- **脅威と脆弱性管理**: 継続的な脆弱性スキャンと優先順位付き修復
- **Microsoft Defender for Cloud**: マルチクラウド環境全体の統一された保護

#### サプライチェーン攻撃への対策

サプライチェーン侵害は、信頼されたサードパーティのソフトウェアやサービスを経由して発生します。

- **ソフトウェアコンポジション分析**: オープンソースの依存関係の脆弱性を検出
- **CI/CD パイプラインのセキュリティ**: GitHub Advanced Security による脆弱性スキャン
- **サードパーティリスクの評価**: ベンダーのセキュリティ態勢を評価するプロセス
- **ゼロトラストの適用**: 信頼されたサードパーティのアクセスも常に検証

#### AI 固有のリスクへの対策

AI システムの導入に伴う固有のセキュリティリスクへの対策:

- **責任ある AI の原則**: 公平性、信頼性、プライバシー、セキュリティを確保
- **Microsoft Purview AI Hub**: AI によるデータアクセスと使用状況の可視化
- **プロンプトインジェクション対策**: AI への悪意ある入力攻撃への防御
- **AI 出力の検証**: AI が生成するコンテンツの適切なフィルタリング

## モジュール 4: ランサムウェアやその他の攻撃に対する回復性戦略の設計

### 学習目標

- ランサムウェアなどの一般的なサイバー脅威を理解する
- 組織のビジネス回復性を支援する戦略を設計できる
- セキュアなバックアップと復元のための構成を設計できる
- セキュリティ更新プログラムを管理するためのソリューションを設計できる

---

### 4-1. ランサムウェアの理解

#### ランサムウェアとは

ランサムウェアは、被害者のファイルを暗号化し、復号のための身代金を要求するマルウェアです。現代のランサムウェア（人間が操作する Human-Operated Ransomware）は、以下の特徴を持ちます。

- **人間による操作**: 自動化されたマルウェアではなく、攻撃者が手動で操作する
- **事前調査**: 攻撃前に標的の環境を詳細に調査する
- **権限昇格**: 最初の侵入後、管理者権限の取得を試みる
- **横展開 (Lateral Movement)**: ネットワーク内を移動してより多くのシステムに感染させる
- **持続性の確保**: バックドアや追加のマルウェアを設置する
- **データ窃取と恐喝**: データの暗号化だけでなく、窃取したデータの公開を脅迫の手段に使う

#### 典型的な攻撃のフロー

```
ステップ 1: 初期アクセスの取得
  （フィッシング / 脆弱性の悪用 / RDP 侵害 / サプライチェーン攻撃）
              ↓
ステップ 2: 実行と持続性の確保
  （マルウェアの実行 / バックドアの設置）
              ↓
ステップ 3: 権限昇格
  （ローカル管理者 → ドメイン管理者権限の取得）
              ↓
ステップ 4: 横展開
  （ネットワーク内での感染拡大）
              ↓
ステップ 5: データ窃取
  （機密データの外部への送信）
              ↓
ステップ 6: ランサムウェアの展開
  （ファイルの暗号化と身代金の要求）
```

---

### 4-2. ビジネス回復性の支援

#### セキュリティ回復性の概念

セキュリティ回復性とは、攻撃が発生した場合でもビジネスを継続できる能力です。完璧なセキュリティは存在しないという現実を受け入れ、侵害前・侵害中・侵害後のすべてのフェーズにわたって継続的に投資することが重要です。

**回復性戦略の目標:**

| フェーズ | 目標 |
|---|---|
| **侵害前** | 攻撃の成功確率を下げ、被害が発生しにくい環境を作る |
| **侵害中** | 被害の範囲と深刻度を制限し、攻撃を迅速に検出する |
| **侵害後** | 迅速に回復し、通常業務を再開する |

また、ビジネスの観点から回復性は以下を可能にします:

- 急速に変化するビジネス環境への迅速な適応
- 予期しない変化（例: コロナ禍での在宅勤務への突然の移行）への対応
- ビジネスオペレーションへの影響を最小化

---

### 4-3. セキュアなバックアップと復元の設計

#### ランサムウェアに対するバックアップの原則

1. **バックアップの隔離**: 本番環境から論理的または物理的に分離する
2. **定期的な復元テスト**: バックアップからの復元が実際に機能するか定期的に確認する
3. **バックアップ自体の保護**: ランサムウェアがバックアップを暗号化しないよう保護する
4. **RTO/RPO の定義**: 許容できる復旧時間目標 (RTO) と復旧ポイント目標 (RPO) を明確に設定する

#### 3-2-1 バックアップルール

| 数字 | 意味 |
|---|---|
| **3** | データのコピーを 3 つ保持する（本番 1 つ + バックアップ 2 つ） |
| **2** | 異なる 2 種類のストレージメディアに保存する |
| **1** | 1 つはオフサイト（または論理的に分離した場所）に保存する |

#### Azure Backup によるハイブリッドバックアップの保護

| 機能 | 説明 |
|---|---|
| **不変バックアップ** | バックアップデータを削除・変更不可に設定する |
| **論理的な削除 (Soft Delete)** | 誤削除や攻撃による削除を一定期間回復可能にする |
| **バックアップデータの暗号化** | 保存されたバックアップデータをキーで暗号化して保護する |
| **マルチユーザー認証** | 重要な操作（バックアップ無効化など）に複数の承認者を必要とする |
| **プライベートエンドポイント** | バックアップデータへのアクセスをプライベートネットワーク内に制限する |

---

### 4-4. セキュリティ更新プログラムの管理

#### 脆弱性管理の重要性

攻撃者は既知の脆弱性を悪用することが多いため、タイムリーなパッチ適用は重要な防御手段です。

**パッチ管理のベストプラクティス:**

- **資産インベントリの把握**: 組織内のすべての資産とソフトウェアバージョンを把握する
- **脆弱性の優先順位付け**: CVSS スコアや実際の悪用可能性に基づいてパッチを優先する
- **テスト環境での事前検証**: 本番環境への適用前に非本番環境でテストする
- **自動化の活用**: 可能な限りパッチ適用を自動化する
- **例外の補完コントロール**: パッチ適用が困難な場合に代替のセキュリティ対策を実装する

#### Azure Update Manager の活用

Azure Update Manager は、Azure・オンプレミス・マルチクラウドのマシン全体で更新プログラムを一元管理するサービスです。

- 更新コンプライアンス状態の可視化
- スケジュールによる自動更新の設定
- 更新前後のマシン状態の評価
- メンテナンス期間を利用した計画的な更新

---

### 4-5. 特権アクセスのランサムウェア対策

特権アカウントは攻撃者にとって最も重要な標的です。特権アクセスを侵害することで、攻撃者はネットワーク全体にランサムウェアを展開できます。

**特権アクセス保護のベストプラクティス:**

| 対策 | 説明 |
|---|---|
| **特権アクセスワークステーション (PAW)** | 管理作業専用の隔離されたデバイスを使用する |
| **JIT (Just-In-Time) アクセス** | 必要なときだけ特権を付与し、不要時は削除する |
| **多要素認証の強制** | すべての特権アカウントに MFA を必須化する |
| **緊急アクセスアカウント** | 通常の認証が失敗した場合のブレークグラスアカウントを設定する |
| **ドメインコントローラーの保護** | DC へのアクセスを厳しく制限し、特別な保護を適用する |
| **資格情報の衛生管理** | パスワードの再利用を防ぎ、認証情報の漏洩を継続的に監視する |

## まとめ: ラーニングパス全体の重要ポイント

### フレームワーク活用の判断基準

| 状況 | 推奨フレームワーク |
|---|---|
| クラウドへの移行計画を立てる | CAF |
| クラウドソリューションの設計品質を評価する | WAF |
| Microsoft セキュリティ製品の統合アーキテクチャを参照する | MCRA |
| セキュリティコントロールのコンプライアンスを評価する | MCSB |
| ゼロトラストの導入を開始する | ゼロトラスト採用フレームワーク / RaMP |

### 設計時の重要な考慮事項

1. **ゼロトラストを前提に設計する**: 境界防御だけに頼らず、ID を中心にしたセキュリティモデルを採用する
2. **侵害を前提とした設計**: 侵害が発生することを想定し、被害の封じ込めと迅速な回復を設計に組み込む
3. **フレームワークを組み合わせる**: 単一のフレームワークではなく、CAF・WAF・MCRA・MCSB を状況に応じて組み合わせる
4. **継続的な改善**: セキュリティは一度設定すれば終わりではなく、継続的な評価と改善が必要である
5. **ビジネス目標との整合**: セキュリティ要件をビジネス目標と整合させ、過剰なセキュリティによるビジネスの阻害を避ける

### SC-100 試験領域との対応

| 試験領域 | 出題割合 | 対応モジュール |
|---|---|---|
| セキュリティのベストプラクティスと優先順位の設計 | 20〜25% | 本ラーニングパス全体 |
| セキュリティオペレーション・ID・コンプライアンス機能の設計 | 25〜30% | 別ラーニングパス |
| インフラストラクチャのセキュリティソリューションの設計 | 25〜30% | 別ラーニングパス |
| アプリケーションとデータのセキュリティソリューションの設計 | 20〜25% | 別ラーニングパス |
